# 	Câu hỏi quyết định: có extend task không? 
	
1. Có người được assign xử lý không?
2. Có trạng thái lifecycle không? New → In Progress → Closed?
3. Có deadline / SLA không?
4. Có cần comment, work notes, activity log không?
5. Có cần approval không?
6. Có cần queue/list cho agent xử lý không?
7. Có cần report chung với các loại task khác không?
	
		
	Nếu đa số là có → nghiêng mạnh về extend task.

	Nếu đa số là không → đừng extend task.

		Cái này nên cân nhắc extend task.
		Vì bản chất nó là một công việc.	
		
		
		
		
# Rule senior 		
		
			
	Nếu là “work item” chung → xét extend task.
	Nếu là “sự cố service” → incident.
	Nếu là “điều tra root cause” → problem.
	Nếu là “thay đổi hệ thống có risk” → change.
	Nếu là “yêu cầu từ user” → request/catalog.
	Nếu là “tài sản / thiết bị / cấu hình” → asset hoặc CMDB, không phải task.
	Nếu là dữ liệu master/reference → custom/core reference table, không phải task.

# 9. Kết luận thẳng 		
		
		
			
	Đúng, sau này khi thêm loại công việc mới, phải nghĩ đầu tiên: có extend được task không?
		Nhưng không phải vì task là “bảng tốt nhất”.
		Mà vì task là abstraction chuẩn cho work management trong ServiceNow.

	Công thức nhớ:
		Entity/data object → custom table / asset / CMDB
		Work/process item → consider extending task
		Specific ITSM process → incident / problem / change / request

	Sai ở bước này thì sau đó business rule, report, ACL, SLA, flow đều méo theo.
