# 1. Bản chất của Planning & Reasoning là gì?

	Planning = Agent tự chia bài toán lớn thành nhiều bước nhỏ.

	Reasoning = Agent suy nghĩ xem bước nào nên làm trước, bước nào nên làm sau, cần gọi tool nào, rồi tổng hợp kết quả để ra câu trả lời cuối cùng.


	Điểm quan trọng nhất:
		Planning & Reasoning KHÔNG phải AI biết suy nghĩ như con người.


	Nó chỉ là:
		phân tích mục tiêu
		tạo kế hoạch
		thực hiện từng bước
		nếu cần thì điều chỉnh kế hoạch
		cuối cùng trả lời.

	Trong AWS Bedrock Agent, việc này được LLM thực hiện dựa trên:
		Prompt
		Instructions
		Danh sách Tool (Action Group)
		Knowledge Base
		Context


	Agent không được lập trình cứng kiểu
		Step 1
		↓

		Step 2
		↓

		Step 3

		mà nó tự quyết định.
		Đây là điểm khác biệt lớn nhất.



# 2. Ví dụ đời thường

	Giả sử bạn hỏi:
		Tôi muốn đi Osaka cuối tuần sau với ngân sách 30,000 yên.

	Một chatbot bình thường sẽ chỉ trả lời:
		Bạn nên đi Shinkansen.

	Hết.



	Agent thì khác.
		Nó sẽ tự lập kế hoạch.
		Ví dụ trong đầu Agent sẽ là:

		Mục tiêu:
			↓
			Tìm chuyến đi Osaka
			↓
			Xác định ngày
			↓
			Kiểm tra giá vé
			↓
			Kiểm tra khách sạn
			↓
			Tính tổng tiền
			↓
			So sánh với ngân sách
			↓
			Đề xuất lịch trình



		Không ai lập trình các bước này.
			LLM tự tạo.
			Đó gọi là
			Planning.


		Sau đó
			Agent tiếp tục suy luận.

		Ví dụ:
			Nếu khách sạn quá đắt
			↓
			Chọn khách sạn khác
			↓
			Nếu vẫn vượt ngân sách
			↓
			Đổi sang xe bus
			↓
			Nếu vẫn vượt
			↓
			Giảm số đêm


		Đó gọi là
			Reasoning.



# 3. Ví dụ trong System Design
	Giả sử công ty có AI Support.
	User hỏi:

		Khách hàng A đã thanh toán chưa?
		Nếu rồi thì gửi hóa đơn.
		Nếu chưa thì gửi email nhắc.


	Agent sẽ không có API duy nhất.
		Nó sẽ tự nghĩ.


		Goal
		↓
		Check Customer
		↓
		Call CRM API
		↓
		Đã thanh toán?
		↓
		YES
		↓
		Call Invoice API
		↓
		Send Email
		--------------------

		NO
		↓
		Reminder API
		↓
		Send Email



		Không có flow nào được code cứng.
			Agent quyết định.




# 4. Mapping với AWS

| Thành phần                         | Vai trò                    |
| ---------------------------------- | -------------------------- |
| Amazon Bedrock Agent               | Thực hiện Planning         |
| Foundation Model (Claude, Nova...) | Suy luận                   |
| Action Group                       | Các Tool có thể gọi        |
| Lambda                             | Thực hiện Tool             |
| Knowledge Base                     | Cung cấp dữ liệu           |
| Prompt                             | Quy định cách lập kế hoạch |
| Session Memory                     | Ghi nhớ context            |


	Planning & Reasoning thường đi cùng các dịch vụ sau:
	Ví dụ.
	
		User
		↓
		Bedrock Agent
		↓
		Reasoning
		↓
		Need Tool?
		↓
		Action Group
		↓
		Lambda
		↓
		Database
		↓
		Return
		↓
		Reasoning
		↓
		Need More Tool?
		↓
		YES
		↓
		Call Again
		↓
		Final Answer



# 5. Multi-step Reasoning 
	Đây là keyword rất hay xuất hiện.


	Ví dụ.
		User hỏi
		Tổng doanh thu tháng này của các khách hàng VIP ở Tokyo là bao nhiêu?

		Agent sẽ chia thành nhiều bước.
			Step 1
				Lấy danh sách VIP
				↓

			Step 2
				Lọc Tokyo
				↓

			Step 3
				Lấy doanh thu
				↓

			Step 4
				Cộng
				↓

			Step 5
				Tạo báo cáo

		Không cần developer viết từng bước.

# 6. Dynamic Planning 
	Điểm mạnh của Agent.
	Kế hoạch có thể thay đổi.

		Tool A lỗi
		↓
		Agent thử Tool B
		↓
		Tool B lỗi
		↓
		Knowledge Base
		↓
		Không có
		↓
		Xin lỗi user



	Không cần
		if else
		do developer viết.




# 7. ReAct Pattern 
	AWS Bedrock thực tế dùng ý tưởng gần giống
	ReAct (Reason + Act).
	
	Chu trình:
		Reason
		↓
		Action
		↓
		Observation
		↓
		Reason
		↓
		Action
		↓
		Observation
		↓
		Final Answer


	Ví dụ.
		Reason
		"Tôi cần giá cổ phiếu"
		↓
		Action
		Call API
		↓
		Observation
		"200 USD"
		↓
		Reason
		"Tính lợi nhuận"
		↓
		Action
		Calculator
		↓
		Observation
		Done
		↓
		Answer

		Đây chính là cách đa số AI Agent hiện đại hoạt động.
		
		
		
# 8. Trade-off 		
		
| Ưu điểm                          | Nhược điểm                                 |
| -------------------------------- | ------------------------------------------ |
| Không cần lập trình flow cố định | Có thể suy luận sai                        |
| Linh hoạt                        | Chi phí Token cao hơn                      |
| Dễ mở rộng                       | Latency lớn hơn                            |
| Thích hợp workflow phức tạp      | Debug khó hơn                              |
| Tự chọn Tool                     | Kết quả không phải lúc nào cũng giống nhau |
		
		
		
# 9. So sánh dễ nhầm 		
		
| Workflow Engine      | AI Agent                       |
| -------------------- | ------------------------------ |
| Step cố định         | Step động                      |
| Developer quyết định | LLM quyết định                 |
| Flow không đổi       | Flow thay đổi                  |
| Logic nằm trong code | Logic nằm trong Prompt + Model |
| Deterministic        | Probabilistic                  |
