# 1. Action Group là gì? 

	Action Group chính là "bộ tay chân" (Execution Layer) của Agent.
	Nếu Knowledge Base giúp Agent biết, thì Action Group giúp Agent làm được.


	Nói ngắn gọn:
		LLM chỉ biết suy nghĩ.
		Action Group giúp biến suy nghĩ thành hành động.



	Ví dụ:
		Người dùng hỏi
		"Đặt vé máy bay cho tôi."
		LLM hiểu ý.

	Nhưng LLM không thể:
		gọi API của Vietnam Airlines
		ghi Database
		gửi Email
		thanh toán Visa

	Agent phải dùng Action Group.


	Có thể tưởng tượng:
		User
		↓
		Agent
		↓
		LLM suy nghĩ
		↓
		Action Group
		↓
		Lambda
		API Gateway
		REST API
		MCP Server
		Database
		CRM
		ERP
		...
		↓
		Kết quả
		↓
		LLM giải thích lại cho User

	Đây chính là mô hình Reason → Act → Observe → Respond.


# 2. Tại sao phải có Action Group?
	LLM có giới hạn rất lớn.


	Ví dụ:
		Người dùng hỏi
		"Kiểm tra đơn hàng #12345"
		Claude biết phải làm gì.


	Nhưng Claude KHÔNG BIẾT:
		Database ở đâu
		Password là gì
		Endpoint là gì
		API nào dùng

		LLM không được phép tự truy cập hệ thống nội bộ.
		Action Group chính là cầu nối.


	Ví dụ thực tế

		Khách hỏi

			Đơn hàng của tôi đang ở đâu?
			↓
			Claude hiểu
			↓
			Action Group
			↓
			GET /orders/12345
			↓
			API trả về
			Shipping
			↓
			Claude
			Đơn hàng đang được giao.



# 3. Action Group gồm những gì?
	Thông thường gồm:
		Action Name
		Description
		Input Schema
		Authentication
		Target

	Ví dụ

		Action Name
			GetOrder

		Description
			Lấy thông tin đơn hàng

		Input
			orderId
	
		Target
			Lambda

		Agent đọc Description để biết khi nào cần gọi.



# 4. Agent quyết định gọi Action như thế nào? 


Đây là phần rất nhiều người hiểu sai.
	Không phải:
		User
		↓
		Lambda

	Mà là

		User
		↓
		Claude
		↓
		Reasoning
		↓
		Có cần gọi tool không?
		↓
		YES
		↓
		Action Group
		↓
		Lambda
		↓
		Kết quả
		↓
		Claude
		↓
		User

		LLM chính là người quyết định.
		Giống ChatGPT quyết định khi nào dùng Search.

		Ví dụ

		User:
			Hôm nay thời tiết thế nào?

		Claude:
			Không cần Action.



		User:
			Đặt lịch họp.
	
		Claude:
			Cần Action.
			↓
			Call Calendar API




# 5. Action Group gọi Lambda

		User

		↓

		↓

		Ví dụ

			Retry

		Đây là best practice của AWS.

# 8. Action Group gọi MCP Server 

	Đây là hướng mới của AI.
	MCP
	(Model Context Protocol)
	Không gọi API riêng lẻ nữa.
	Thay vào đó


	Agent
	↓
	MCP Server
	↓
	Tool 1
	Tool 2
	Tool 3

	Filesystem
	GitHub
	Slack
	Database

	...
	
	
		MCP giống USB-C.
		Một chuẩn.
		Không cần viết API riêng từng Tool.
			
					
		Ví dụ
		Agent hỏi
		Tạo Issue GitHub


		Không cần biết
		POST /issues
		headers
		token

		...

		Chỉ cần
		create_issue()
		MCP Server xử lý hết.

# 13. Trade-off


| Cách          | Ưu điểm                   | Nhược điểm                         |
| ------------- | ------------------------- | ---------------------------------- |
| Lambda Direct | Đơn giản                  | Khó chia sẻ API                    |
| API Gateway   | Quản lý tốt, bảo mật      | Tăng thêm một lớp, có chi phí      |
| REST API      | Tích hợp mọi hệ thống     | Phải tự xử lý auth, retry, version |
| MCP Server    | Chuẩn AI mới, mở rộng tốt | Hệ sinh thái còn đang phát triển   |

# 15. Các bẫy đề thi AWS

| Bẫy                                          | Đáp án đúng                                                                                           |
| -------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| Knowledge Base có thực hiện hành động không? | ❌ Không, chỉ truy xuất tri thức.                                                                      |
| LLM có tự gọi API không?                     | ❌ Không, phải thông qua Action Group/Tool.                                                            |
| Action Group có phải Business Logic không?   | ❌ Không, nó chỉ là lớp định nghĩa và điều phối việc gọi công cụ. Business Logic nằm trong Lambda/API. |
| API Gateway có bắt buộc không?               | ❌ Không. Có thể gọi Lambda trực tiếp nếu phù hợp.                                                     |
| MCP có thay thế Lambda không?                | ❌ Không. MCP là giao thức chuẩn hóa việc truy cập công cụ; Lambda vẫn có thể là công cụ được MCP gọi. |

	AWS thường không hỏi:
		"Action Group là gì?"

	Mà sẽ đưa scenario.

	Ví dụ:

	Một Bedrock Agent cần:
		trả lời tài liệu nội bộ
		tạo ticket ServiceNow
		gọi hệ thống ERP

	Cần dùng gì?
	
		Nếu chỉ cần đọc tài liệu:
			→ Knowledge Base

		Nếu cần thao tác với hệ thống:
			→ Action Groups

		Nếu cần thực thi logic:
			→ Lambda hoặc API

		Nếu cần quản lý API tập trung:
			→ API Gateway

		Đây là chuỗi suy luận mà AWS mong đợi thí sinh nhận ra.
