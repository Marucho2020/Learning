# 1. Tag là gì? (Bản chất) 
	Tag là metadata (siêu dữ liệu) được gắn vào Resource để giúp con người quản lý Resource.


	Nói đơn giản:
		Tag không làm thay đổi cách Resource hoạt động.
		Nó chỉ thêm "thông tin mô tả" để AWS hoặc con người biết Resource đó thuộc ai, dùng làm gì, tính tiền cho ai...

	Giống như dán nhãn lên đồ đạc.


	Ví dụ:

	Bạn có 100 chiếc laptop.
	Nếu không dán nhãn:
		Laptop 1
		Laptop 2
		Laptop 3
		...
		Laptop 100

	Bạn sẽ không biết:
		của phòng nào
		của nhân viên nào
		dùng cho dự án gì

	Nếu dán tag:
		Owner = Nguyen
		Department = HR
		Project = CRM
		Environment = Production

	thì việc quản lý cực kỳ dễ.
	AWS cũng hoạt động giống hệt như vậy.


# 2. Tag gồm những gì?

	Một Tag luôn là
		Key = Value


	Ví dụ
		Project = Payment
		Environment = Production
		Owner = TeamA
		CostCenter = IT001
		AWS không hiểu "Project" nghĩa là gì.


	AWS chỉ lưu:
		Project
		↓
		Payment
	Chính doanh nghiệp tự quy định ý nghĩa.

# 	4. Tag dùng để làm gì?

	Đây là phần AWS cực kỳ thích hỏi.

| Mục đích          | Có dùng Tag không? |
| ----------------- | ------------------ |
| Quản lý Resource  | ✅                  |
| Phân chia chi phí | ✅                  |
| IAM Policy        | ✅                  |
| Automation        | ✅                  |
| Backup            | ✅                  |
| Billing Report    | ✅                  |
| Tìm kiếm          | ✅                  |



	## 4.1 Cost Allocation
		Ví dụ
		AWS bill
		$200,000
		CEO hỏi
		Team AI tiêu bao nhiêu?
		Nếu không Tag
		Không biết.
		Nếu Tag
		
		CostCenter = AI
		CostCenter = HR
		CostCenter = Sales

		AWS Cost Explorer sẽ thống kê
			AI
			$70,000
			HR
			$40,000
			Sales
			$90,000

		Đây gọi là
		Cost Allocation Tags
		Rất hay thi.



	## 4.2 IAM 
		Có thể giới hạn quyền dựa trên Tag.
		Ví dụ
		Developer A
		chỉ được sửa

			Project = AI
		không được động vào
			Project = Banking
			IAM Policy sẽ kiểm tra
			aws:ResourceTag/Project
			Đây là ABAC (Attribute-Based Access Control).



	## 4.3 Automation

			Ví dụ

			Environment = Dev
			Lambda chạy mỗi tối
			↓
			Shutdown toàn bộ EC2 có
			Environment = Dev
			để tiết kiệm tiền.
			Không cần liệt kê từng server.



	## 4.4 Backup
			AWS Backup có thể backup
			Backup = Daily
		hoặc
			Backup = Weekly
			Thay vì chọn từng EC2.



	## 4.5 Organization

			Công ty lớn
			100 AWS Accounts
			50000 resources
			Tag là cách duy nhất để quản lý.

# 6. Tag KHÔNG làm được gì?


	Đây là bẫy cực kỳ phổ biến.

	Tag KHÔNG:
		❌ tăng CPU

		❌ tăng RAM

		❌ tăng Security

		❌ tăng Availability

		❌ tăng Performance

		❌ thay đổi Network

		❌ thay đổi Storage

	Tag chỉ là

	Metadata


# 8. So sánh dễ nhầm

| Khái niệm   | Bản chất                      |
| ----------- | ----------------------------- |
| Name        | Chỉ là tên hiển thị           |
| Tag         | Metadata gồm nhiều Key-Value  |
| Resource ID | AWS tạo, duy nhất             |
| ARN         | Định danh đầy đủ của Resource |

# 9. Trade-off

		Ưu điểm
			Quản lý cực dễ
			Cost Allocation
			Automation
			IAM
			Governance
			Reporting

		Nhược điểm
			Phải thống nhất quy tắc đặt Tag.
			Thiếu Tag sẽ khó quản lý.
			Sai Tag có thể khiến Automation, Backup hoặc IAM áp dụng nhầm.

# 11. Cách AWS ra đề



	Dạng 1
		"Công ty muốn biết mỗi phòng ban tiêu bao nhiêu tiền."
		Đáp án
		✅ Cost Allocation Tags

	Dạng 2
		"Developer chỉ được quản lý EC2 của chính nhóm mình."
		Đáp án
		✅ IAM + Resource Tags (ABAC)

	Dạng 3
		"Muốn backup toàn bộ server Production."
		Đáp án
		✅ AWS Backup + Tag
		Không phải chọn từng EC2.

	Dạng 4
		"Muốn tự động tắt toàn bộ Development EC2 vào ban đêm."
		Đáp án
		✅ Lambda/EventBridge + Tag

	Dạng 5 (Bẫy)
		"Tag giúp tăng hiệu năng quản lý EC2."
		Sai.
		Tag chỉ hỗ trợ quản lý, không cải thiện hiệu năng.
