# 🔥 1. BẢN CHẤT CỦA SCAN

	Scan = đọc toàn bộ bảng (full table scan)
		- Không quan tâm partition key
		- Không có index → vẫn đọc hết
		- Sau đó lọc (FilterExpression) chỉ là bước sau cùng

	👉 Nghĩa là:

		❌ Bạn filter 1 record
		✅ Nhưng DynamoDB vẫn đọc cả triệu record trước đã




# ⚠️ 2. Ví dụ thực tế 

	Case: hệ thống user

		Table: Users
			PK: userId
			Có 1 field: age


	❌ Sai lầm phổ biến

		Bạn muốn lấy:
			age > 30

		Bạn dùng Scan:
			Scan + FilterExpression: age > 30


	👉 Điều xảy ra:
		DynamoDB đọc toàn bộ table (ví dụ 1 triệu record)
		Sau đó mới filter ra (ví dụ còn 10k record)

	💥 Bạn bị:
		Tốn RCU cực lớn
		Latency cao
		Scale kém

	✅ Cách đúng
		Tạo GSI:
		PK: age
		→ dùng Query



# ⚡ 3. Mapping với AWS services


| Concept                  | AWS Service          |
| ------------------------ | -------------------- |
| Full scan DB             | DynamoDB Scan        |
| Efficient lookup         | DynamoDB Query       |
| Secondary access pattern | GSI / LSI            |
| Batch processing         | Scan + Parallel Scan |





# 🧠 4. TỪ KHÓA TRONG ĐỀ THI 


	Nhìn thấy mấy từ này → nghĩ ngay Scan:

		"read entire table"
		"no specific key"
		"analyze all data"
		"data migration"
		"ETL job"
		"one-time operation"
		"low frequency access"

	👉 Ngược lại:

		"real-time query"
		"high performance"
		"user request"

	→ ❌ Không bao giờ dùng Scan




# ⚠️ 5. BẪY AWS RẤT HAY GÀI


	❌ Trap 1: FilterExpression đánh lừa
		“Use Scan with FilterExpression to reduce cost”

	👉 Sai hoàn toàn
		Filter không giảm cost read
		Chỉ giảm data trả về




	❌ Trap 2: Small data nên Scan ok

		👉 Sai tư duy
			Đề thi assume scale lớn
			Không bao giờ design system production với Scan



	❌ Trap 3: Scan vs Query confusion

| Scan           | Query              |
| -------------- | ------------------ |
| đọc toàn bộ    | đọc theo key       |
| chậm           | nhanh              |
| tốn RCU        | tiết kiệm          |
| dùng cho batch | dùng cho real-time |



# ⚖️ 6. Trade-off


| Scan tốt khi     | Scan tệ khi    |
| ---------------- | -------------- |
| ETL              | user API       |
| migration        | real-time      |
| analytics batch  | high traffic   |
| data exploration | cost sensitive |


	👉 Kết luận thẳng:
		Scan chỉ dùng cho backend job, không bao giờ dùng cho API người dùng




# 🧪 7. Scenario giống đề thi

	🧩 Question

		Một hệ thống cần:

		Phân tích toàn bộ dữ liệu mỗi đêm
		Không yêu cầu real-time
		Dataset lớn

	👉 Chọn gì?
		✅ Answer
		→ Scan (có thể kết hợp Parallel Scan)

	🧩 Question (bẫy)

		Một API cần:
			trả dữ liệu theo condition
			response nhanh
			traffic cao

			❌ Sai
				→ Scan + Filter

			✅ Đúng
				→ Query + GSI


# 🎯 8. CÁCH AWS RA ĐỀ
	AWS sẽ KHÔNG hỏi trực tiếp "Scan là gì"

	👉 Nó sẽ hỏi kiểu:

		Pattern 1: disguise
		“Application needs to evaluate all items in a table”
		→ Scan

	Pattern 2: trap performance
		“low latency required”
		→ loại Scan ngay


	Pattern 3: cost optimization

		“reduce read capacity usage”
		→ không chọn Scan



#🧠 9. So sánh Scan vs Query 

| Tiêu chí     | Scan        | Query         |
| ------------ | ----------- | ------------- |
| Key required | ❌           | ✅             |
| Performance  | ❌           | ✅             |
| Cost         | ❌           | ✅             |
| Use case     | batch       | real-time     |
| Filter       | sau khi đọc | trước khi đọc |

# 🔥 1. Scanの本質

	Scan = テーブル全体を読み取る（フルスキャン）

		パーティションキー不要
		インデックス不要
		FilterExpressionは後処理

	👉 つまり：
		❌ 1件だけ欲しい
		✅ でも全件読んでからフィルタ


# ⚠️ 2. 実例 

	Usersテーブル：
		userId（PK）
		age

	❌ 間違い
		age > 30


	Scan + FilterExpression

		👉 結果：
			全データを読み取る
			その後フィルタ

		💥 問題：
			RCU大量消費
			レイテンシ高い





	✅ 正しい方法 

		GSI作成：
		PK: age
		→ Query使用




# ⚡ 3. AWSサービスとの対応

| 概念        | AWS           |
| --------- | ------------- |
| 全件取得      | Scan          |
| 高速検索      | Query         |
| 別アクセスパターン | GSI           |
| バッチ処理     | Parallel Scan |



# 🧠 4. 試験キーワード 

	Scanを示すワード：
		- 全データ取得
		- 分析
		- ETL
		- 一括処理
		- 低頻度アクセス


	Scanを避けるワード：
		- リアルタイム
		- 高速レスポンス
		- API処理


# ⚠️ 5. よくある罠

	❌ Filterでコスト削減できる？
		→ できない


	❌ 小さいデータならOK？
		→ 試験ではNG



	❌ Queryとの混同
		| Scan | Query |
		| ---- | ----- |
		| 全件   | Key指定 |
		| 遅い   | 速い    |
		| 高コスト | 低コスト  |



# ⚖️ 6. トレードオフ 

	Scanが適するケース：

		- ETL
		- バッチ分析
		- データ移行

	不適切：

		- API
		- リアルタイム処理




# 🧪 7. 試験シナリオ 

	問題
		夜間バッチで全データ分析
		→ ✅ Scan


	問題（罠）

		高速API必要
		→ ❌ Scan
		→ ✅ Query + GSI



# 🎯 8. AWSの出題パターン 

	- 「全件評価」→ Scan
	- 「低レイテンシ」→ Scan排除
	- 「コスト削減」→ Scan排除


# 🧠 9. まとめ 

	👉 Scanは：
		「全部読むしかない時の最終手段」
