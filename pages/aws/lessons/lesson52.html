Lesson 52 -- Cloud Practitioner --PK--SK -//


# 🔥 1. PK (Partition Key) là gì — nói thẳng

		PK = cách DynamoDB chia dữ liệu ra nhiều máy (partition)


		👉 Nó quyết định:
			Data nằm ở đâu
			Load được phân tán ra sao

		👉 Nếu PK thiết kế ngu:
		→ tất cả request dồn vào 1 chỗ → hot partition → choke → throttling



	## Ví dụ thực tế
		PK = userId
		👉 Mỗi user → 1 vùng dữ liệu riêng
			→ 1 triệu user → data được spread ra nhiều partition → scale tốt




	## ❌ Sai lầm chết người 
		PK = "ORDER"


		👉 Tất cả data cùng 1 PK
		→ dồn vào 1 partition
		→ hệ thống chết ngay khi có load



# ⚙️ 2. SK (Sort Key) là gì — bản chất

	SK = cách sắp xếp + filter dữ liệu bên trong 1 PK

		👉 Nó cho phép:
			Query theo range
			Sort
			Lấy subset data

		PK = userId
		SK = createdAt

			👉 Query:

			Lấy order mới nhất
			Lấy order trong khoảng thời gian


		## ❗ Bản chất sâu hơn
		PK → chọn “bucket”
		SK → tìm “trong bucket”

		👉 PK = WHERE
		👉 SK = ORDER BY + RANGE




# 🧠 3. Hiểu theo kiểu senior

	PK = X  (bắt buộc)
	+ optional SK condition
		👉 Không có PK → KHÔNG query được (chỉ có scan)


# 🔥 4. Ví dụ real system 


	Bài toán:
		User có nhiều order
		Cần lấy order gần nhất


		PK = userId
		SK = timestamp

		userId = A
		order by timestamp desc
		limit 1

		→ O(1), cực nhanh



# ⚠️ 5. Bẫy cực hay gặp 

		❌ Bẫy 1: nghĩ SK là optional
			👉 Sai
				Không có SK → mất:
				sorting
				range query
				flexibility

		❌ Bẫy 2: nhét nhiều data vào 1 PK
			PK = "GLOBAL"
				→ chết vì hot partition



		❌ Bẫy 3: SK không có meaning
			SK = randomUUID

			→ không sort được
			→ mất hết lợi ích


# ⚖️ 6. So sánh nhanh



| Thành phần | Vai trò thật                  |
| ---------- | ----------------------------- |
| PK         | phân phối data (scale)        |
| SK         | tổ chức data (query trong PK) |


# 🎯 7. Cách AWS ra đề

		Nó không hỏi:
			PK là gì?
		
		Nó hỏi kiểu:
			👉 “Design table for efficient query by user and time”
			→ phải tự hiểu:
				PK = userId
				SK = time


# 🧠 8. Insight level senior (cái này quan trọng) 


	DynamoDB KHÔNG phải database bình thường
		Nó là:

	👉 key-value + sorted storage
		👉 PK = chọn shard
		👉 SK = sorted index trong shard


# ⚡ 9. Tư duy đúng

	Khi design:
		❌ “Data của mình có gì?”
		✅ “Mình query cái gì?”
