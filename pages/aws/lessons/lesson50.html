# I. Throughput là gì (nói thẳng, không vòng vo)

	Throughput = lượng dữ liệu hệ thống xử lý được trong 1 đơn vị thời gian

	👉 Không phải “tốc độ” chung chung
	👉 Không phải “latency”
	👉 Là khối lượng công việc/giây



	Ví dụ:
		1000 request/giây
		500 MB/s đọc từ disk
		10,000 records/giây xử lý

	👉 Công thức tư duy:
		Throughput = Work / Time



# II. Bản chất (level senior)

	Throughput không đứng một mình. Nó luôn trade-off với 2 thứ:


| Yếu tố      | Ý nghĩa                    |
| ----------- | -------------------------- |
| Latency     | 1 request mất bao lâu      |
| Concurrency | bao nhiêu request cùng lúc |



	👉 Hiểu sâu:
		Throughput ≈ Concurrency / Latency


		👉 Nghĩa là:
			Muốn throughput cao:
			Tăng concurrency (scale out)
			Giảm latency (optimize)

		👉 AWS exam cực thích đánh vào đây.





# III. Ví dụ thực tế (system design)



	Scenario 1 – Web API

	Bạn có API:
		xử lý 1 request = 100ms
		chạy 100 thread

	👉 Throughput:
		= 100 / 0.1s = 1000 request/sec





	Scenario 2 – DynamoDB

			1 item = 1KB
			1000 writes/sec

		👉 Throughput:
			= 1000 KB/sec = 1MB/sec

		👉 Đây chính là:
			Write Capacity Units (WCU)





# IV. Mapping với AWS Services


	## 1. Amazon DynamoDB 
		Throughput = Capacity

| Loại | Ý nghĩa          |
| ---- | ---------------- |
| RCU  | read throughput  |
| WCU  | write throughput |



		👉 Bản chất:
			Bạn đang mua throughput trước




	## 2. Amazon S3 
		👉 S3 không nói trực tiếp “throughput”
		Nhưng thực chất:

		request/sec
		MB/sec

		👉 Key insight:

		S3 scale automatically
		Không cần provision throughput




	## 3. Amazon EBS 
	
			👉 Có 2 thứ:

			IOPS (số request)
			Throughput (MB/s)

			👉 Ví dụ:

			gp3:
			3000 IOPS
			125 MB/s throughput

			👉 Trap:

			IOPS cao nhưng throughput thấp vẫn choke	



	## 4. Amazon Kinesis 

		👉 Throughput = shard capacity
			1 shard:
			1 MB/s write
			2 MB/s read

		👉 Muốn scale → tăng shard



# V. Từ khóa hay xuất hiện trong đề

	high throughput
	low throughput bottleneck
	provisioned throughput
	burst throughput
	throughput scaling
	throttling



# VI. Bẫy (trap) cực kỳ hay gặp 


| Sai                          | Đúng |
| ---------------------------- | ---- |
| Throughput cao = nhanh       | Sai  |
| Throughput cao = xử lý nhiều | Đúng |
	👉 1 request có thể chậm nhưng hệ thống vẫn throughput cao





	##Trap 2 – DynamoDB throttling

		👉 Nếu:

			vượt quá RCU/WCU

		→ bị:
			throttling

		👉 Keyword:
			ProvisionedThroughputExceededException





	## Trap 3 – EBS IOPS vs Throughput 

		👉 AWS rất thích hỏi:
			workload nhỏ nhiều request → cần IOPS
			workload file lớn → cần throughput

		👉 Sai lầm phổ biến:
			chọn sai loại volume




	## Trap 4 – Kinesis shard 

		👉 Không hiểu:
		throughput tied với shard
			→ scale sai



# VII. So sánh dễ nhầm

| Concept    | Ý nghĩa             | Khi dùng            |
| ---------- | ------------------- | ------------------- |
| Throughput | lượng xử lý/giây    | big data, streaming |
| Latency    | thời gian 1 request | realtime            |
| IOPS       | số operation/giây   | DB / disk           |

# 定義（本質）			
			
			
	Throughputとは：

	👉 単位時間あたりに処理できるデータ量
			リクエスト数/秒
			MB/秒			
					
			
	本質理解			
			Throughput ≈ 同時処理数 / レイテンシ
							
			同時処理を増やす → Throughput増加
			レイテンシを下げる → Throughput増加			
						
					
					
					
# AWSサービスとの対応 					
			
			
		Amazon DynamoDB
			RCU / WCU = Throughputそのもの
			超えると throttling 発生			
					
			
		Amazon S3
			自動スケーリング
			Throughputを意識しなくてよい			
			
			
		Amazon EBS
			IOPS vs Throughput
			大きなファイル → Throughput重視	
						
			
		Amazon Kinesis
			1 shard = Throughput単位
			shard追加でスケール			
			
			
			
# よく出るキーワード 			
			
		high throughput
		throttling
		provisioned throughput
		scaling
