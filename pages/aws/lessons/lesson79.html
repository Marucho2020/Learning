# 1. Bản chất của EventBridge (Định nghĩa không mơ hồ)
	Amazon EventBridge là một Event Bus (xe buýt sự kiện) dùng để kết nối các hệ thống với nhau theo mô hình Event-Driven Architecture.

	Nói đơn giản:
		Một service phát ra Event → EventBridge nhận → EventBridge quyết định chuyển Event đó đến service nào.


	EventBridge không xử lý business logic, nó chỉ làm nhiệm vụ:
		Nhận Event
		Lọc Event
		Định tuyến (Routing)
		Gửi đến đúng Target

	Nó giống như:
		Bưu điện thông minh cho Event.



	Ví dụ:
		EC2 vừa được tạo.

	EC2 phát Event:
		EC2 Instance State Change


	↓
		EventBridge nhận được

	↓

	Rule:
		Nếu event = EC2 running

	↓

	Forward sang
		Lambda
		SNS
		SQS
		Step Functions
		ECS
		Kinesis
		API Destination

	...

	Tất cả đều tự động.



# 2. Event là gì?

	Event chính là:
		Một thông báo rằng "đã có chuyện gì xảy ra".


Ví dụ:

	Object uploaded vào S3
	User login
	EC2 stopped
	RDS backup completed
	CodePipeline failed
	CloudTrail ghi log
	Schedule 8AM
	Custom Application gửi message



	Tất cả đều là Event.

	Một Event thường có dạng JSON

	Ví dụ

	{
	  "source": "aws.ec2",
	  "detail-type": "EC2 Instance State-change Notification",
	  "detail": {
		  "state":"running"
	  }
	}

	EventBridge đọc JSON này rồi quyết định gửi đi đâu.

# 4. Event Bus là gì?


	Rất nhiều người nhầm EventBridge và Event Bus.

Thực ra:
	EventBridge là dịch vụ.

	Bên trong nó có:
		Event Bus

	Giống như
		Post Office
		↓
		Có nhiều hộp thư


	Event Bus chính là nơi chứa Event tạm thời để routing.

Có 3 loại

Loại	Dùng khi nào
Default Event Bus	AWS Service gửi Event
Custom Event Bus	Application của bạn gửi Event
Partner Event Bus	SaaS như Datadog, Zendesk, Auth0...



# 5. Rule là gì?

	Rule giống IF.

	Ví dụ

		IF
		source == aws.ec2
		AND
		state == stopped
		THEN
		Invoke Lambda


		Rule không viết code.
		Chỉ cấu hình điều kiện.




# 6. Target là gì?
	Target là nơi Event cuối cùng được gửi tới.
		Lambda
		SNS
		SQS
		ECS
		Step Functions
		API Destination
		CloudWatch Logs
		Batch
	Target chính là nơi business logic chạy.



Ví dụ 1

	Một công ty lưu ảnh lên S3.
	Yêu cầu

		Khi upload ảnh
		↓
		Resize ảnh
		↓
		Lưu thumbnail
		Kiến trúc
		User
		↓
		S3
		↓
		EventBridge
		↓
		Lambda
		↓
		Thumbnail

	Không cần polling.
	Không cần cron.
	Hoàn toàn event-driven.



	Ví dụ 2

		EC2 bị Stop.

			EC2
			↓
			EventBridge
			↓
			SNS
			↓
			Email Admin

		Không cần server theo dõi.





	Ví dụ 3

	Mỗi sáng 8 giờ

		Schedule
		↓
		EventBridge
		↓
		Lambda
		↓
		Generate Report

	EventBridge thay thế Cron Job.

# 9. So sánh với SNS và SQS

|                   | EventBridge   | SNS                  | SQS               |
| ----------------- | ------------- | -------------------- | ----------------- |
| Mục đích          | Event Routing | Fan-out Notification | Queue             |
| Có Filter         | ✅ Rất mạnh    | Có nhưng hạn chế hơn | ❌                 |
| Có Schedule       | ✅             | ❌                    | ❌                 |
| Routing theo JSON | ✅             | Hạn chế              | ❌                 |
| Lưu Queue         | ❌             | ❌                    | ✅                 |
| Pull Message      | ❌             | ❌                    | ✅                 |
| Event-driven      | ✅             | ✅                    | Không phải router |



	Bản chất

		SNS

			Publish
			↓
			Broadcast

		EventBridge

			Publish
			↓
			Rule
			↓
			Routing

		SQS

			Publish
			↓
			Queue
			↓
			Consumer



# 10. Trade-off

	Ưu điểm
		Decouple hệ thống rất tốt.
		Không cần polling.
		Dễ mở rộng.
		Hỗ trợ hàng trăm nguồn sự kiện AWS và SaaS.
		Routing dựa trên nội dung JSON.

	Nhược điểm
		Không phải hàng đợi lưu trữ lâu dài như SQS.
		Không thay thế Kafka/Kinesis cho streaming tốc độ rất cao.
		Chỉ nên dùng để điều phối sự kiện, không chứa business logic.
