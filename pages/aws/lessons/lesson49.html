# 1. Định nghĩa 

	DynamoDB là:
		Distributed database managed hoàn toàn, scale ngang tự động, 
		latency millisecond ổn định, 
		nhưng đánh đổi bằng việc bạn phải thiết kế schema theo query pattern

	👉 Nói dễ hiểu:
		Không có JOIN
		Không query linh hoạt như RDB
		Muốn query gì → phải design key trước


# 2. Core concept (cái này mới là thứ AWS hỏi) 

| Thành phần    | Bản chất                               |
| ------------- | -------------------------------------- |
| Partition Key | Quyết định data nằm ở node nào (scale) |
| Sort Key      | Sắp xếp trong partition                |
| Partition     | Nơi chứa data thật (scale ngang)       |
| RCU / WCU     | Đơn vị throughput                      |
| GSI           | Index để query pattern khác            |
| Scan vs Query | Scan = ngu, Query = đúng               |




3. Ví dụ thực tế (system design) 

	❌ Sai (dev kiểu RDB mindset)

		User Table
		- id
		- name

		Order Table
		- user_id
		- order_id


		→ Bạn muốn:
			lấy tất cả order của user
			→ DynamoDB: KHÔNG JOIN → fail




	✅ Đúng (DynamoDB mindset)
		PK = USER#123
		SK = ORDER#2026-01-01

		👉 Query:
			PK = USER#123

		👉 Đây gọi là:
			Single Table Design




# 4. Mapping AWS Service 

| Use case             | AWS                 |

| Real-time API        | DynamoDB            |

| Complex query        | ❌ RDS               |

# 5. Từ khóa trong đề thi 


	- single-digit millisecond latency
	- massive scale
	- serverless
	- eventually consistent
	- strongly consistent (optional)
	- partition key design
	- hot partition
	- GSI / LSI
	- on-demand vs provisioned



# 6. Bẫy (trap AWS hay gài) 

	❌ Trap 1: chọn DynamoDB cho query phức tạp
		"user muốn filter nhiều field linh hoạt"
		→ sai → phải chọn RDS


	❌ Trap 2: hot partition
		PK = "USER"
		→ tất cả request đập vào 1 partition → chết


	❌ Trap 3: Scan
		Scan = đọc toàn bộ table → tốn tiền + chậm
		

	❌ Trap 4: consistency
		Default = eventual
		Không hiểu → chọn sai trong hệ thống tài chính


# 7. So sánh dễ nhầm

| Feature  | DynamoDB        | RDS                  |
| -------- | --------------- | -------------------- |
| Scale    | Auto horizontal | Vertical             |
| Query    | Limited         | Flexible SQL         |
| Join     | ❌               | ✅                    |
| Latency  | ~ms             | cao hơn              |
| Use case | high scale API  | transactional system |



# 8. Trade-off (cái senior phải hiểu) 

| Ưu           | Nhược            |
| ------------ | ---------------- |
| Scale vô hạn | design khó       |
| serverless   | query hạn chế    |
| latency thấp | không join       |
| HA mặc định  | phải denormalize |

# 1. 定義（本質）
	フルマネージドの分散データベースであり、自動的にスケールし、
	ミリ秒レベルの低レイテンシを提供するが、クエリパターンに基づいた設計が必須


	👉 要するに：
		JOINできない
		柔軟な検索は苦手
		事前にアクセスパターンを設計する必要がある

# 2. コア概念 

| 項目            | 本質                 |
| ------------- | ------------------ |
| Partition Key | データの配置とスケールを決定     |
| Sort Key      | パーティション内の順序        |
| Partition     | 実際のデータ保存単位         |
| RCU / WCU     | スループット単位           |
| GSI           | 別の検索パターン用インデックス    |
| Query vs Scan | Queryは効率的、Scanは非効率 |



# 3. 実例

	❌ RDB思考（NG）
		JOIN前提 → DynamoDBでは不適



	✅ 正しい設計
		PK = USER#123
		SK = ORDER#日時

		→ Queryで一発取得
			👉 Single Table Design

6. よくあるトラップ

	❌ 複雑検索に使う

	❌ Scan使用
		→ コスト増・遅い
