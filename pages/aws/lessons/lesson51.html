# 🔥 1. Định nghĩa bản chất 

	Query Pattern = cách bạn thiết kế data để đọc (read), KHÔNG phải cách bạn ghi (write).

	❗ Sai lầm phổ biến: nghĩ database lưu xong rồi query kiểu gì cũng được 
		→ sai hoàn toàn trong NoSQL (DynamoDB)


	Trong AWS (đặc biệt là DynamoDB):

		👉 Bạn phải biết trước:
			Query gì?
			Query theo field nào?
			Tần suất ra sao?

			→ rồi thiết kế table theo query đó


# ⚙️ 2. Ví dụ thực tế (system design) 

	Scenario:
		Bạn build hệ thống ecommerce.

		Yêu cầu:
			Lấy danh sách order của user
			Sort theo thời gian mới nhất

	❌ Thiết kế sai (SQL mindset)
		Query:
			SELECT * FROM Orders WHERE userId = 'A' ORDER BY createdAt DESC


		👉 Với DynamoDB:
			Không có index → phải Scan toàn table
			Cost cao + chậm → fail




	✅ Thiết kế đúng (Query-driven)
		Partition Key: userId
		Sort Key: createdAt

			userId = A
			ORDER BY createdAt DESC

		→ DynamoDB query O(1), cực nhanh


# ☁️ 3. Mapping với AWS 

| Pattern         | AWS Service           |
| --------------- | --------------------- |
| Key-based query | DynamoDB (Query API)  |
| Full scan       | DynamoDB Scan (avoid) |
| Secondary query | DynamoDB GSI / LSI    |
| Complex query   | Aurora / RDS          |
| Search flexible | OpenSearch            |

# ⚠️ 5. Bẫy (trap AWS rất hay gài) 

		❌ Trap 1: Scan vs Query
			Scan: đọc toàn bộ table
			Query: đọc theo key

		👉 Đề sẽ viết kiểu:
			“Retrieve data efficiently”

		→ Đáp án:
		❌ Scan
		✅ Query



		❌ Trap 2: Thiết kế giống SQL
		Nhiều table + join
		👉 DynamoDB:

			❌ Không join
			✅ Denormalization


		❌ Trap 3: Thêm index sau
		
			Dev hay nghĩ:

				“Sau này thêm GSI là xong”

				👉 Sai:

				GSI tốn tiền
				Limit throughput
				Design từ đầu vẫn quan trọng

# ⚡ 9. Trade-off



| Design                     | Ưu điểm      | Nhược điểm          |
| -------------------------- | ------------ | ------------------- |
| Query-optimized (DynamoDB) | cực nhanh    | phải design trước   |
| Flexible (RDS)             | query tự do  | scale kém hơn       |
| GSI nhiều                  | flexible hơn | tốn tiền + phức tạp |




# 🧠 10. Kết luận (level senior)

	DynamoDB không phải DB → nó là query engine dựa trên key
	Nếu không biết query trước → design sẽ fail
	Query pattern = bản thiết kế hệ thống, không phải chi tiết nhỏ

# 🔥 1. 本質の定義 

	Query Pattern = データの「読み方」に基づいた設計（書き方ではない）


		❗よくある誤解：
		「データを保存してから好きに検索できる」
		→ DynamoDBでは完全に間違い

		👉 先に決めるべき：

		どのように取得するか
		どのキーで検索するか
		頻度はどうか

		→ それに基づいてテーブル設計を行う


# ⚙️ 2. 実例


	ECサイト
	要件：
		ユーザーの注文一覧取得
		時系列順で並べる


	❌ 悪い設計（SQL思考）
			Orders:
			- orderId
			- userId
			- createdAt



	✅ 正しい設計
		Partition Key: userId
		Sort Key: createdAt	
		→ Queryで高速取得可能
