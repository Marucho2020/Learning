# 1. Amazon RDS — BẢN CHẤT THỰC SỰ 

	✅ Định nghĩa bản chất (nói thẳng)
		RDS KHÔNG phải chỉ là database.

	👉 Nó là:
		Stateful compute + storage + replication system được AWS quản lý

		Tức là:
			Compute: instance (CPU/RAM để chạy DB engine)
			Storage: EBS
			Replication: sync / async giữa AZ

		👉 Quan trọng:
			Compute scale ≠ Storage scale (có thể tách riêng)
			Failover = chuyển compute, không phải storage


	⚠️ Insight mà đề thi hay gài
		RDS không phải “auto scale vô hạn”
		
		RDS vẫn bị giới hạn bởi:
			- IOPS
			- connection limit
			- vertical scaling




# 2. 🧠 ARCHITECTURE NỘI BỘ (THỨ MÀ AWS KHÔNG VIẾT THẲNG)

	## Multi-AZ (HA)

	👉 Bản chất:
		Synchronous replication
		1 Primary + 1 Standby (khác AZ)

		Write → Primary → replicate sync → Standby → commit

		👉 Kết luận:

		✔ Không mất dữ liệu
		❌ Tăng latency write
		❌ Không scale read




# Read Replica (Scale đọc) 

	👉 Bản chất:
		Asynchronous replication

	👉 Flow:
		Primary → async → Replica


	👉 Kết luận:
		✔ Scale read
		✔ Có thể cross-region
		❌ Có delay (eventual consistency)



# 🔥 So sánh cực kỳ quan trọng


| Feature      | Multi-AZ | Read Replica |
| ------------ | -------- | ------------ |
| Mục đích     | HA       | Scale read   |
| Replication  | Sync     | Async        |
| Data loss    | Không    | Có thể       |
| Read traffic | ❌        | ✔            |
| Failover     | Auto     | Manual       |


	👉 90% câu hỏi thi xoay quanh bảng này




# 3. ⚙️ PERFORMANCE — THỨ SENIOR PHẢI HIỂU 

	## Storage type 

| Type    | Use case  |
| ------- | --------- |
| gp2/gp3 | general   |
| io1/io2 | high IOPS |

	👉 Trap:
		“High throughput” → chọn io1/io2





	## Bottleneck thật sự
		Disk IOPS
		Network
		Connection limit

		👉 Không phải CPU là chính


# 4. 📌 USE CASE THỰC TẾ (SYSTEM DESIGN) 

	## 🧪 Scenario 1 — ERP / Banking 


		Yêu cầu:
			Transaction mạnh
			Consistency cao
			Quan hệ dữ liệu phức tạp

		👉 Solution:
			RDS + Multi-AZ

		👉 Vì:
			Không chấp nhận mất dữ liệu
			ACID là bắt buộc





	## 🧪 Scenario 2 — Web app nhiều read 
	
	
	Yêu cầu:
		- 90% read
		- Query phức tạp
		- Scale read

		👉 Solution:
			RDS + Read Replica

		👉 Pattern:
			Write → Primary
			Read → Replica
		
	
	
	
	
	
	
	## 🧪 Scenario 3 — Global app
	
	
		Yêu cầu:
			User nhiều region
			Latency thấp

		👉 Solution:
			RDS + cross-region Read Replica	
			
			
	
	
	## 🧪 Scenario 4 — Burst traffic 
	
		👉 Sai lầm phổ biến:
			“Scale RDS để handle traffic”

		👉 Sai.

		👉 Đúng:
			Cache (ElastiCache)
			Read replica
			Queue	
				
	
# 5. 🔗 MAPPING AWS (THỰC TẾ KIẾN TRÚC)

| ------------------- | ----------------- |
| HA                  | Multi-AZ          |

# 	7. ⚠️ TRAP
	
| Bẫy                             | Thực tế                |
| ------------------------------- | ---------------------- |
| Multi-AZ = scale                | ❌                      |
| Read Replica = HA               | ❌                      |
| RDS auto scaling = như DynamoDB | ❌                      |
| Failover = không downtime       | ❌ vẫn có vài chục giây |
| Read replica luôn consistent    | ❌                      |




# 8. ⚖️ TRADE-OFF

| Ưu điểm            | Nhược điểm      |
| ------------------ | --------------- |
| Strong consistency | Scale kém       |
| SQL linh hoạt      | Cost cao        |
| Managed            | Giới hạn tuning |


# 9. ⚔️ SO SÁNH NHẦM LẪN

| Concept                  | Khác nhau            |
| ------------------------ | -------------------- |
| Multi-AZ vs Read Replica | HA vs Scale          |
| RDS vs DynamoDB          | SQL vs NoSQL         |
| RDS vs Aurora            | Aurora scale tốt hơn |

# 1. Amazon RDS — 本質 


	✅ 定義

		RDSは単なるDBではない。

	👉 本質：
		状態を持つコンピュート＋ストレージ＋レプリケーションの管理システム
		Compute（DBインスタンス）
		Storage（EBS）
		Replication（同期・非同期）

	⚠️ 重要ポイント	
		無限スケールではない
		IOPS・接続数がボトルネック	



# 2. アーキテクチャ 


	## Multi-AZ 
	
			同期レプリケーション
			自動フェイルオーバー

			👉 特徴：
				データ損失なし
				書き込み遅延あり
				読み取りスケール不可



	## Read Replica 

			非同期レプリケーション

		👉 特徴：
			読み取りスケール可能
			遅延あり（Eventually Consistent）


	## 比較 

| 項目       | Multi-AZ | Read Replica |
| -------- | -------- | ------------ |
| 目的       | HA       | 読み取りスケール     |
| 同期       | Sync     | Async        |
| フェイルオーバー | 自動       | 手動           |
| 読み取り     | 不可       | 可能           |




# 3. パフォーマンス

	IOPSが重要
	ストレージタイプ：
		gp3（一般）
		io1/io2（高性能）


# 4. ユースケース 

	例1：金融システム
		→ RDS + Multi-AZ

	例2：読み取り多いWeb
		→ RDS + Read Replica

	例3：グローバル
		→ Cross-region Replica


# 5. マッピング 

| 問題     | 解決           |
| ------ | ------------ |
| 高可用性   | Multi-AZ     |
| 読み取り負荷 | Read Replica |
| 接続数問題  | RDS Proxy    |
