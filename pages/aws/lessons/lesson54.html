# Ad-hoc Query là gì trong AWS

	Ad-hoc query = truy vấn phát sinh tức thời để trả lời một câu hỏi cụ thể mà chưa biết trước.


	Nói thẳng:
		Không phải query cố định chạy hàng ngày
		Không phải batch job định kỳ
		Không phải ETL pipeline chuẩn hóa sẵn

	Là kiểu:
		“Tôi cần tìm dữ liệu này NGAY bây giờ để điều tra / phân tích / debug / audit.”

	Bản chất thực sự:
		Ad-hoc query tồn tại vì dữ liệu ngày nay quá lớn → không thể precompute mọi thứ.


	Cho nên AWS cung cấp các dịch vụ:
		query trực tiếp trên data lake
		không cần dựng database trước
		không cần ETL toàn bộ trước
		trả tiền theo lượng data scan

	Đây là triết lý cực kỳ quan trọng của modern analytics trên AWS.

	##Ví dụ 1 — Điều tra lỗi production

# Vì sao AWS thích ra Athena trong ad-hoc query?
	Vì đây là keyword cực mạnh:

| Requirement       | Service |
| ----------------- | ------- |
| Serverless query  | Athena  |
| Query S3 directly | Athena  |
| Pay per query     | Athena  |
| No infrastructure | Athena  |
| ANSI SQL          | Athena  |
	Chỉ cần thấy combo này → gần như Athena.

# Các bẫy AWS hay gài

	## Bẫy 1 — Redshift vs Athena

| Athena             | Redshift             |
| ------------------ | -------------------- |
| Query trực tiếp S3 | Data warehouse       |
| Ad-hoc nhẹ         | Analytics lớn        |
| Serverless         | Cluster              |
| Không ingest nhiều | Thường cần load data |
| Pay per scan       | Pay cluster          |


	## AWS trap

	Đề ghi:
		“occasionally query”
		“infrequent”
		“S3 data”
		“minimal operational overhead”

	→ Athena.
		Nếu chọn Redshift → sai vì overkill.



	## Bẫy 2 — EMR vs Athena 

| Athena           | EMR                 |
| ---------------- | ------------------- |
| SQL đơn giản     | Big data processing |
| Serverless       | Cluster management  |
| Ad-hoc analytics | Heavy Spark/Hadoop  |



	Trap
	Nếu đề chỉ cần:
		simple SQL
		operational simplicity
	→ Athena.

	EMR chỉ đúng khi:
		custom Spark job
		ML pipeline
		petabyte-scale processing phức tạp




	## Bẫy 3 — RDS query

	Nhiều người nghĩ:
		“Query thì dùng database.”
		Sai trong cloud architecture hiện đại.

	Nếu:
		dữ liệu ở S3
		chỉ query thỉnh thoảng
		analytics
		→ Athena tốt hơn RDS rất nhiều.

	Vì:
		không cần provision DB
		scale tự động
		cost thấp




# Trade-off thực tế


| Athena mạnh                  | Athena yếu                     |
| ---------------------------- | ------------------------------ |
| Không quản lý server         | Chậm hơn DB tuned              |
| Scale cực dễ                 | Query nhỏ nhiều lần có thể tốn |
| Tốt cho S3 data lake         | Không hợp OLTP                 |
| Cost thấp cho sporadic query | Scan nhiều dữ liệu sẽ đắt      |



# Bản chất senior-level cần hiểu

	Athena KHÔNG phải database.

	hơn là MySQL/PostgreSQL.
