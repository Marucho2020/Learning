CMDB（Configuration Management Database）

--------- CMDB（Configuration Management Database）----


# ① ELI5
	Hãy tưởng tượng bạn có một cuốn sổ ghi lại:

		Nhà bạn có những gì (TV, tủ lạnh, router…)
		Mỗi cái nằm ở đâu
		Cái nào đang hỏng
		Cái nào liên quan với nhau (router hỏng → Wi-Fi chết)

	👉 CMDB = cuốn sổ tổng hợp tất cả tài sản IT + mối quan hệ của chúng



# ② Trong dự án thực tế (ServiceNow / SaaS) 
	CMDB trong thực tế không phải “database cho có”, mà là trung tâm của toàn bộ ITSM

	## Dùng như thế nào: 
		- Incident: check CI nào bị lỗi
		- Change: biết hệ thống nào sẽ bị ảnh hưởng
		- Problem: phân tích root cause
		- Asset: quản lý lifecycle



	## Ví dụ thực tế trong ServiceNow: 
	
		Table: cmdb_ci, cmdb_ci_server, cmdb_ci_appl
		Quan hệ: Depends on, Runs on, Connected to	
			
	
	## Câu Nhật hay dùng: 
	
		CMDBを最新状態に保つ必要があります
		CI間の依存関係をCMDBで管理する
		CMDBの精度が低いと影響分析ができない	
		
		👉 Nói thẳng: CMDB bẩn = system mù		
	
	
	
# 	③ Level senior (bản chất hệ thống)
	
	## CMDB giải quyết 1 vấn đề cực lớn:
		👉 “Hệ thống của mình thực sự gồm những gì và chúng liên quan thế nào?”


	## Nếu không có CMDB: 

		- Change → deploy xong production chết
		- Incident → fix sai server
		- Impact analysis → đoán mò
		- Audit → fail


	## Bản chất: 
		CMDB = graph của hệ thống IT

		Node = CI
		Edge = relationship	
		
			👉 Đây không phải database thường → nó là knowledge graph
	
	
	## Liên kết với concept khác:
		
		可視化（Visualization） → nhìn thấy toàn bộ system
		一元化（Centralization） → 1 nguồn sự thật duy nhất
		標準化（Standardization） → cấu trúc CI thống nhất
		ガバナンス（Governance） → kiểm soát thay đổi	
			
			
# Ví dụ tài liệu (IT Nhật) 	
	
		本システムでは、構成管理情報をCMDBにて一元管理する。
		各CI間の依存関係を定義することで、変更時の影響範囲を可視化し、
		インシデント対応および問題管理の精度向上を図る。	
			


# Slide

		CMDBにて構成情報を一元管理
		CI間の関係性を明確化
		影響分析の精度向上
		運用効率の改善



# Meeting phrase 

		「今回の変更で影響範囲、CMDB上で確認済みですか？」
		「CMDBの情報が古いので、正確な分析ができません」

---------------------- CI（Configuration Item）------------------	
			
			
# 	① ELI5
	CI = một thứ cụ thể trong hệ thống IT		
			
	Ví dụ:	
		1 server
		1 database
		1 app
		1 API

		👉 Nói đơn giản:
			CI = 1 “cục” trong hệ thống			
			
			
			
# 	② Trong dự án thực tế		
	CI là đơn vị nhỏ nhất trong CMDB		
			
			
		Ví dụ:
			Web server → 1 CI
			DB server → 1 CI
			Load balancer → 1 CI			
					
					
					
# Trong ServiceNow: 			
			
					
		cmdb_ci_server
		cmdb_ci_database
		cmdb_ci_appl			
			
			
			
# Nhật hay nói: 			
			
		CIを登録する
		CIの関係性を定義する
		CIのステータスを更新する			
					
			
# Use case thực tế: 

		Incident:
			“Server A down” → CI: Server A

		Change:
			“Update DB” → CI: DB server

			
			
		
# ③ Level senior (bản chất) 


	CI là abstraction:

		👉 “đơn vị quản lý cấu hình”
		Không phải chỉ là hardware:
		Có thể là:
			Service
			API
			Business app




	## Vấn đề CI giải quyết: 

		Tracking cấu hình
		Version control (config level)
		Mapping dependency


	## Nếu không có CI: 
		
		Không biết cái gì bị ảnh hưởng
		Không trace được issue
		Không quản lý change			


		👉 CMDB = graph
		👉 CI = node trong graph
		

		各システム構成要素をCIとして定義し、CMDBに登録する。
		CI間の関係性を明確化することで、障害発生時の影響範囲を迅速に特定可能とする。


# Slide

		- CI単位で構成要素を管理
		- CMDBに登録
		- 関係性を定義
		- 影響範囲分析に活用



# Meeting phrase 


		「このCIってどのシステムに依存していますか？」
		「対象CIはCMDBに登録されていますか？」


# So sánh & bẫy 
	## CI vs Resource


| CI                | Resource   |
| ----------------- | ---------- |
| Logical + managed | Raw        |
| Có relationship   | Không      |
| ITSM dùng         | Infra dùng |



	## CI vs Component

		Component → technical part
		CI → managed unit

# Kết nối CMDB vs CI 


	CI = data
	CMDB = structure + relationship

	👉 Nếu bạn chỉ có CI mà không có relationship
	→ CMDB vô dụng

	👉 Nếu bạn có relationship nhưng CI sai
	→ hệ thống phân tích sai
