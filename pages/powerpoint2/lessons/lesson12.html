# 1. Incident（インシデント / 事故対応）

	## ① ELI5
		Hệ thống bị lỗi → người dùng không dùng được → phải sửa NGAY.

		Ví dụ:
			WiFi công ty chết → không làm việc được → gọi IT sửa
		👉 Incident = “cái đang cháy”



	## ② Trong dự án thực tế

		Dùng trong:
			- Service Desk
			- Monitoring alert
			- User ticket
			
		Ví dụ flow:
			User báo:
				「システムにログインできません」
				Service Desk tạo Incident
				Assign cho team
				Fix → Close


	## Câu hay dùng: 
		「インシデント対応を優先してください」
		「SLA内で解決が必要です」

	👉 Trong ServiceNow:

		Table: incident
		Có SLA, priority, assignment group



	##③ Level senior (bản chất)
		Incident = symptom (triệu chứng), không phải nguyên nhân
		
		Nó giải quyết:
			Downtime
			User impact

		Không có Incident:
			- Không tracking được lỗi
			- Không đo SLA
			- Không biết hệ thống đang “cháy ở đâu”


		Liên kết:
			- 可視化 (visibility): biết bao nhiêu lỗi đang xảy ra
			- SLA管理 (SLA management)
			- 運用効率 (operation efficiency)

		👉 Senior mindset:
			Incident chỉ là “dập lửa”, không phải “chữa bệnh”



		## Ví dụ tài liệu 

			ユーザー影響を最小化するため、インシデント管理プロセスを導入し、
			発生から解決までの対応時間をSLAに基づき管理する。



		Slide
			インシデント管理導入
				ユーザー影響の最小化
				SLAによる対応時間管理
				一元的なチケット管理

		Meeting phrase
			「この件はインシデントとして即時対応が必要です」



		So sánh & bẫy
			❌ Sai của junior:
				Nghĩ Incident = bug fix
				👉 Sai.

| Concept  | Meaning        |
| -------- | -------------- |
| Incident | lỗi xảy ra     |
| Bug      | lỗi trong code |

		NG:
			「このバグはインシデントです」
		OK:
			「この障害はインシデントとして管理します」



		## Mindset senior 
			Dùng trong:
				運用 (operation)
				障害対応 (incident response)
				
			Không dùng trong:
				設計 (design)
				改善提案

		👉 Khi viết tài liệu:
			Nếu nói về “lỗi đang xảy ra” → Incident
			Nếu nói về “nguyên nhân” → Problem



# 2. Problem（問題管理 / 根本原因）

	## ① ELI5

		Cái lỗi xảy ra nhiều lần → phải tìm lý do thật sự

		Ví dụ:
			WiFi hay chết → phát hiện router lỗi
			👉 Problem = “tại sao nó cháy?”




	## ② Trong dự án thực tế

		Flow:
			- Incident xảy ra nhiều lần
			- Tạo Problem
			- Root Cause Analysis (RCA)
			- Đề xuất fix lâu dài

		Ví dụ câu:
			「根本原因を特定するため、Problemを起票します」
			「恒久対策を検討します」

		Trong ServiceNow:
			Table: problem
			Có liên kết với nhiều Incident


	## ③ Level senior (bản chất)
		Problem = root cause abstraction	
	
	
		Nó giải quyết:
			- Lỗi lặp lại
			- Technical debt

		Không có Problem:
			- Incident sẽ lặp vô hạn
			- System “chữa cháy mãi mãi”	
			
		Liên kết:

			- 根本原因分析 (RCA)
			- 改善 (continuous improvement)
			- ナレッジ蓄積 (knowledge accumulation)	
				
		👉 Senior hiểu:
			Nếu chỉ có Incident mà không có Problem → system chết dần
	
	
	## Ví dụ tài liệu 
		同一インシデントの再発防止のため、Problem管理を実施し、
		根本原因の分析および恒久対策を策定する。	 (Lập ra)


	## Slide 
		Problem管理
			- 根本原因の特定
			- 再発防止
			- ナレッジ化	
	
	
	
	## Meeting phrase 
		「このインシデントはProblemとして分析した方がいいです」	
			
	
	
	
	
	## So sánh & bẫy	
	
| Concept  | Meaning     |
| -------- | ----------- |
| Incident | triệu chứng |
| Problem  | nguyên nhân |
	
	
	
	❌ Junior sai:
		Tạo Problem cho mọi Incident

	👉 Sai. Chỉ tạo khi:
		Lặp lại
		Impact lớn
		Unknown root cause	
			
			
		NG:
			「すべてProblem化する」
			
		OK:
			「再発性のあるもののみProblem化する」			
					
			
			
			
	## Mindset senior 			
			
		Dùng trong:
			改善フェーズ
			分析フェーズ
			
		Không dùng trong:
			Immediate fix

		👉 Senior luôn nghĩ:
			Có đáng để phân tích không? Hay fix xong là đủ?			
					
					
	
	
	
# 	3. Change（変更管理 / リリース）
	
	## ① ELI5 
			Muốn sửa hệ thống → phải xin phép → test → deploy
		Ví dụ:

			Đổi router WiFi → phải báo trước → tránh làm sập hệ thống
			👉 Change = “đụng vào hệ thống phải kiểm soát”



	## ② Trong dự án thực tế

			- Tạo Change Request
			- Review (CAB – Change Advisory Board)
			- Approve
			- Deploy


		Câu hay dùng:
			「本対応はChangeとして申請してください」
			「リリース影響を確認してください」


		Trong ServiceNow:
			Table: change_request




	## ③ Level senior (bản chất)
	
		Change = controlled risk

		Mọi thay đổi đều có risk:
			- downtime
			- bug mới
			- conflict


		Không có Change:
			Dev deploy bừa → production chết


		Liên kết:
			リスク管理 (risk management)
			標準化 (standardization)
			ガバナンス (governance)


		👉 Senior hiểu:
			Change không phải để làm chậm, mà để tránh phá hệ thống
			
	
	
	## Ví dụ tài liệu
	
		本番環境への影響を最小化するため、変更管理プロセスを導入し、
		すべてのリリースを事前承認制とする。	
			
	
	
	## Slide
		Change管理
			- リリース統制
			- リスク低減
			- 事前承認プロセス
	
	
	
	## Meeting phrase
		「この対応はChange申請が必要です」
	
	
	
	## So sánh & bẫy 
| Concept | Meaning           |
| ------- | ----------------- |
| Change  | thay đổi hệ thống |
| Release | deploy            |
	
	
	
	
	❌ Junior sai:
			Nghĩ Change = deploy

			👉 Sai:

			Change = process
			Release = action
	
	NG:
		「リリース＝Change」
		
	OK:
		「Change承認後にリリース実施」	
	
	
	## Mindset senior 

		Dùng trong:
			- 設計
			- 運用
			- ガバナンス
			
		Không dùng:
			- Fix nhỏ local (dev env)

		👉 Senior luôn hỏi:

			- Impact là gì?
			- Rollback được không?
			- Ai approve?


# Tổng kết cực thẳng

| Concept  | Bản chất                   |
| -------- | -------------------------- |
| Incident | đang cháy                  |
| Problem  | tại sao cháy               |
| Change   | sửa sao cho không cháy nữa |


👉 Nếu bạn nói chuyện như senior:

	Incident → “対応”
	Problem → “分析”
	Change → “統制”
