# 	1. 検証結果 (result） 
	
	Ý nghĩa (trong IT project)

		Không phải “kết quả chung chung”
		→ Là output có căn cứ từ test / PoC / kiểm chứng kỹ thuật	
			
		👉 Nhật họ đọc cái này để quyết định:
			- Có deploy không
			- Có scale không
			- Có risk không		
		
	
	Ví dụ thực tế (ServiceNow)
	
		- Kiểm chứng workflow approval khi concurrent request
		- Test performance khi 1000 request/phút
		- Kiểm tra integration với external API	
			
	Đoạn tài liệu (report style)		
	
		本検証では、ServiceNowの承認ワークフローに対して同時リクエストを送信し、
		処理性能および競合発生有無を確認した。
		その結果、同時100件までは遅延なく処理可能であることを確認したが、
		200件を超えた場合、一部の承認処理において最大5秒の遅延が発生した。	
					
		👉 Điểm mấu chốt:
			Có điều kiện (100件, 200件)
			Có số liệu (5秒)
			Có kết luận rõ	
				
	
	
	##Slide (PowerPoint) 

		
		
		- ServiceNow承認WFの同時実行検証を実施
		- 100件までは遅延無しで処理可能
		- 200件以上で最大5秒の遅延発生
		- 高負荷時のパフォーマンス劣化を確認
	
	## Title chuẩn Nhật 	

	👉 承認ワークフローの同時実行に関する検証結果	
	
	
	##	NG vs OK
	
		❌ NG (junior)
			検証の結果、問題は特にありませんでした。
	
		👉 Sai ở:
			Không số liệu
			Không điều kiện
			Không ai tin
	
	
		✅ OK (senior)	
			同時100件までの処理において性能劣化が確認されなかったが、200件以上では最大5秒の遅延が発生した。
	



# 2. 制約（limitation） 

	Ý nghĩa (trong IT project)
		Không phải “hạn chế chung chung”
		→ Là điều kiện hệ thống KHÔNG thể vượt qua

	👉 Đây là thứ:
		Dev phải biết để design
		PM phải biết để tránh overpromise



	## Ví dụ thực tế (ServiceNow / SaaS) 

		- API rate limit (1000 req/min)
		- Không xử lý real-time
		- Không hỗ trợ transaction multi-record
			
			
		本システムでは、ServiceNowの標準APIを利用して外部連携を行うため、
		API呼び出し回数は1分あたり最大1000回の制限がある。
		そのため、大量データの一括処理においては、バッチ分割またはキューイング処理の導入が必要となる。			
					
				
			👉 Điểm mấu chốt:
				Nêu rõ constraint
				Nêu impact
				Nêu hướng xử lý			
						
			
	## Slide (PowerPoint)		
			
			- ServiceNow標準APIにより外部連携を実施
			- API呼び出しは最大1000回/分の制限あり
			- 大量処理時は制限超過リスクあり
			- バッチ分割またはキュー処理が必要			
						
			
	## Title chuẩn Nhật 		
			👉 外部連携におけるAPI利用制約について
			
			
			
	## NG vs OK 
	
		❌ NG (junior)		
			
		APIに制限があります。		
			👉 Vô dụng. Không ai dùng được.
			
			
		✅ OK (senior)	
			API呼び出しは1分あたり最大1000回に制限されているため、大量データ処理時にはバッチ分割が必要となる。
			
			
# Tổng kết (đừng hiểu sai) 			
			
	検証結果 = “Sự thật đã test → dùng để quyết định”
	制約 = “Giới hạn hệ thống → bắt buộc phải sống chung”			
				
	👉 Sai lầm phổ biến của dev:
		Viết 検証結果 như cảm nhận
		Viết 制約 mà không gắn với impact

	👉 Level senior:
		Luôn có số liệu + điều kiện + hậu quả + hướng xử lý

	## Context (giống genba thật)

	Bạn đang làm dự án ServiceNow導入 cho một công ty.
			- Có chức năng: 承認ワークフロー（approval workflow）
			- Có integration với hệ thống ngoài qua API
			- Bạn đã làm:
				- 負荷テスト (load test)
				- API連携テスト



	## Yêu cầu 

		👉 Viết 1 đoạn tài liệu (10–15 dòng) bao gồm:

		1. 検証結果
			Kết quả load test workflow
			Có:
			Số lượng request (ví dụ: 100件, 500件…)
			Có hoặc không có delay
			Có hoặc không có lỗi
			
		2. 制約
			API external có giới hạn (tự giả định số)
			Phải:
			Nêu rõ limit
			Nêu impact
			Nêu hướng xử lý (quan trọng)

		■検証結果(result)

			##  Đoạn văn  

			本検証では、ServiceNowの標準承認ワークフローに対して同時実行負荷テストを実施し、
			処理性能および競合発生有無を確認した。
			その結果、1分あたり300件までの同時実行においては遅延およびエラーは発生しなかったが、
			400件以上では最大30秒の遅延および一部処理エラーが発生することを確認した。
		
		
	
			##  PowerPoint 
				- ServiceNowの標準承認フローにより同時に実行検証を実施
				- 300件/分までは遅延無で処理可能
				- 400件/分以上は30秒の遅延および処理エラー発生
				- 高負荷時のパフォーマンス劣化を確認


		■制約
		
			##  Đoạn văn  
			本システムではServiceNow標準のAPIを利用して外部連携を行うため、
			API呼び出し回数は1分あたり最大1000回の制限がある。
			そのため、大量データの一括処理においてはバッチ分割およびキューイングの導入が必要となる。
			
			
			本システムでは、ServiceNowの標準APIを用いた外部連携を行うため、
			API呼び出し回数は1分あたり最大1000回に制限されている。
			そのため、大量データを一括処理した場合、API制限超過によりリクエストエラーが発生するリスクがある。
			この対策として、バッチ分割およびキューイング処理の導入が必要となる。
			
			
			
			##  PowerPoint 
			- ServiceNowの標準APIにより外部連携を実施する
			- API呼び出しは最大1000回/分の制限があり
			- 最大処理時は制限超過のリスクあり
			-　標準APIおよび承認ワークフローはキューイングを利用する必要
			
			
			
			
			


		⚠️ Rule (rất quan trọng)

			Nếu vi phạm → tôi chấm fail thẳng:

		❌ Không được viết mơ hồ kiểu:
			「問題なし」
			「遅延が発生した可能性がある」
		❌ Không được thiếu:
			Số liệu
			Điều kiện
		❌ Không được tách rời:
			制約 mà không có impact → FAIL



	## 🎯 Level tôi expect (rất cao) 
	
		Bạn phải viết kiểu này:
			Có logic:
				なぜ → 何が起きた → だからどうする
	
			Có mindset:
				Dev đọc hiểu được
				PM đọc quyết định được	
		
	🚨 Cách tôi chấm

		Tôi sẽ chấm theo 4 tiêu chí:
			Độ cụ thể (số liệu, điều kiện)
			Logic (có flow hay không)
			Độ “Japanese IT style”
			Tư duy senior (có impact + solution không)	
			
		
#🎯 Câu hỏi của bạn
	
	Tại sao 承認 (workflow) ở 検証結果
		còn API lại ở 制約?
	👉 Vì bản chất 2 thứ này KHÁC NHAU:
		
	## 1. 承認ワークフロー → 検証結果 
	
		👉 Đây là thứ bạn CHỦ ĐỘNG test
			Bạn tạo load
			Bạn đo performance
			Bạn quan sát behavior

		=> Nó tạo ra fact (sự thật đo được)	
			➡️ Vì vậy nó thuộc:
			👉 検証結果 (result)
				
	
	## 2. API制限 → 制約 
	
		👉 Đây là thứ bạn KHÔNG thể thay đổi
			ServiceNow quy định sẵn
			Bạn không control được
			Không cần test cũng biết (doc đã ghi)

		=> Nó là hard limit của system

		➡️ Vì vậy nó thuộc:
		👉 制約 (limitation)



	## ⚠️ Nếu bạn viết ngược lại thì sao? 

		❌ Sai tư duy phổ biến
			Viết API limit vào 検証結果
			→ reviewer sẽ nghĩ:
			👉 “Ơ cái này doc có rồi, test làm gì?”
			
		Viết workflow vào 制約
			→ reviewer sẽ nghĩ:
			👉 “Ơ cái này là behavior, sao lại gọi là limitation?”
