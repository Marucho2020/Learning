# 	1. 検証結果 (result） 
	
	Ý nghĩa (trong IT project)

		Không phải “kết quả chung chung”
		→ Là output có căn cứ từ test / PoC / kiểm chứng kỹ thuật	
			
		👉 Nhật họ đọc cái này để quyết định:
			- Có deploy không
			- Có scale không
			- Có risk không		
		
	
	Ví dụ thực tế (ServiceNow)
	
		- Kiểm chứng workflow approval khi concurrent request
		- Test performance khi 1000 request/phút
		- Kiểm tra integration với external API	
			
	Đoạn tài liệu (report style)		
	
		本検証では、ServiceNowの承認ワークフローに対して同時リクエストを送信し、
		処理性能および競合発生有無を確認した。
		その結果、同時100件までは遅延なく処理可能であることを確認したが、
		200件を超えた場合、一部の承認処理において最大5秒の遅延が発生した。	
					
		👉 Điểm mấu chốt:
			Có điều kiện (100件, 200件)
			Có số liệu (5秒)
			Có kết luận rõ	
				
	
	
	##Slide (PowerPoint) 

		
		
		- ServiceNow承認WFの同時実行検証を実施
		- 100件までは遅延無しで処理可能
		- 200件以上で最大5秒の遅延発生
		- 高負荷時のパフォーマンス劣化を確認
	
	## Title chuẩn Nhật 	

	👉 承認ワークフローの同時実行に関する検証結果	
	
	
	##	NG vs OK
	
		❌ NG (junior)
			検証の結果、問題は特にありませんでした。
	
		👉 Sai ở:
			Không số liệu
			Không điều kiện
			Không ai tin
	
	
		✅ OK (senior)	
			同時100件までの処理において性能劣化が確認されなかったが、200件以上では最大5秒の遅延が発生した。
	



# 2. 制約（limitation） 

	Ý nghĩa (trong IT project)
		Không phải “hạn chế chung chung”
		→ Là điều kiện hệ thống KHÔNG thể vượt qua

	👉 Đây là thứ:
		Dev phải biết để design
		PM phải biết để tránh overpromise



	## Ví dụ thực tế (ServiceNow / SaaS) 

		- API rate limit (1000 req/min)
		- Không xử lý real-time
		- Không hỗ trợ transaction multi-record
			
			
		本システムでは、ServiceNowの標準APIを利用して外部連携を行うため、
		API呼び出し回数は1分あたり最大1000回の制限がある。
		そのため、大量データの一括処理においては、バッチ分割またはキューイング処理の導入が必要となる。			
					
				
			👉 Điểm mấu chốt:
				Nêu rõ constraint
				Nêu impact
				Nêu hướng xử lý			
						
			
	## Slide (PowerPoint)		
			
			- ServiceNow標準APIにより外部連携を実施
			- API呼び出しは最大1000回/分の制限あり
			- 大量処理時は制限超過リスクあり
			- バッチ分割またはキュー処理が必要			
						
			
	## Title chuẩn Nhật 		
			👉 外部連携におけるAPI利用制約について
			
			
			
	## NG vs OK 
	
		❌ NG (junior)		
			
		APIに制限があります。		
			👉 Vô dụng. Không ai dùng được.
			
			
		✅ OK (senior)	
			API呼び出しは1分あたり最大1000回に制限されているため、大量データ処理時にはバッチ分割が必要となる。
			
			
# Tổng kết (đừng hiểu sai) 			
			
	検証結果 = “Sự thật đã test → dùng để quyết định”
	制約 = “Giới hạn hệ thống → bắt buộc phải sống chung”			
				
	👉 Sai lầm phổ biến của dev:
		Viết 検証結果 như cảm nhận
		Viết 制約 mà không gắn với impact

	👉 Level senior:
		Luôn có số liệu + điều kiện + hậu quả + hướng xử lý

	🔥 Bài tập: 検証結果 vs 制約 mâu thuẫn
	Context

		Bạn đang làm ServiceNow integration:
			External API có制約:
				最大1000回/分
		Bạn đã làm load test (検証)
		
		
	💥 Sự thật (cố tình mâu thuẫn)
	
		検証結果 (fact bạn đo được)
			800 req/phút → OK
			1000 req/phút → vẫn OK (không lỗi, không delay)
			1200 req/phút → bắt đầu có lỗi (429)
		
		制約 (doc / vendor nói)
			API limit: 1000 req/phút
		
		
	🎯 Vấn đề

		👉 Theo test:
			1000 vẫn OK
			
		👉 Nhưng theo spec:
			1000 là limit (vượt là chết)

		👉 Mâu thuẫn:
			Tin test?
			Hay tin spec?
			
	📝 Yêu cầu bài làm
		Viết 1 đoạn tài liệu (10–15 dòng) bao gồm:

	1. 検証結果
		Ghi rõ:
		800 / 1000 / 1200
		Behavior từng mức
		
	2. 制約
		Ghi rõ:
		Limit là 1000 (theo spec)
		
		Nhưng:
			Phải chỉ ra mâu thuẫn với test
	3. 判断 / 方針 (QUAN TRỌNG NHẤT)
		👉 Đây là phần phân biệt senior vs junior

	Bạn phải trả lời:
		Có dùng 1000 không?
		Hay phải safe hơn?
		Có cần buffer không?

	
	

	本システムでは、ServiceNow標準APIを用いた外部連携において、API呼び出し性能に関する負荷検証を実施した。
	検証の結果、800リクエスト/分および1000リクエスト/分においては遅延およびエラーは発生しなかったが、
	1200リクエスト/分ではHTTP 429エラーが発生することを確認した。

	一方で、ServiceNowの仕様上、API呼び出しは最大1000リクエスト/分に制限されており、
	検証結果と仕様上の制約に乖離が存在する。
	このため、仕様上の制約を優先しつつ、突発的な負荷増加による制限超過リスクを回避するため、20%のバッファを考慮する。
	その結果、運用上のAPI呼び出し上限は800リクエスト/分とし、
	これを超過しないようバッチ分割およびキューイング制御を実施する方針とする。



	検証結果
		- ServiceNow標準APIを用いた外部連携において負荷検証を実施
		- 800リクエスト/分および1000リクエスト/分では遅延およびエラーは発生しなかった
		- 1200リクエスト/分ではHTTP 429エラーが発生した
		- ※API上限1000リクエスト/分は仕様値であり、検証結果には含めない

	制約
		- ServiceNow仕様により、API呼び出しは最大1000リクエスト/分に制限されている
		- 検証結果と仕様上限に乖離があり、上限付近での運用は制限超過リスクがある
	方針
		- 仕様上の上限（1000リクエスト/分）を遵守する
		- 突発的な負荷増加による制限超過を防ぐため、20%のバッファを確保する
		- 運用上のAPI呼び出し上限は800リクエスト/分とする
		- 上限超過を防止するため、バッチ分割およびキューイング制御を実施する
