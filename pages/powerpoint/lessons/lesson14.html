要件（requirement）
標準機能（standard function）
カスタマイズ


# 1. Ý nghĩa (trong dự án IT)

	■ 要件（ようけん）
	→ “Khách hàng muốn gì” (business + system expectation)
		Không phải solution.
		Sai lầm phổ biến: viết luôn cách làm → lẫn với thiết kế



	■ 標準機能
		→ Những gì hệ thống đã có sẵn (ví dụ: ServiceNow workflow, approval, notification)
		
			Ở Nhật:
			→ ưu tiên 100% standard trước
			→ vì: cost thấp, maintain dễ, upgrade không chết


	■ カスタマイズ
		→ Những gì phải code thêm / chỉnh sửa vì standard không đáp ứng

		Bị ghét nếu lạm dụng
		→ vì: upgrade = dễ vỡ hệ thống




# 2. Ví dụ thực tế (ServiceNow)

	要件
		申請は2段階承認とすること
		承認状況をリアルタイムで確認できること


	判断

| 要件        | 対応               |
| --------- | ---------------- |
| 2段階承認     | 標準Workflowで対応可能  |
| リアルタイム可視化 | 標準Dashboardで対応可能 |
	→ Không cần customize


	Nếu có yêu cầu khó hơn:

		承認条件が複雑（部署＋金額＋役職）
		→ 標準 không đủ → cần カスタマイズ


# 3. Đoạn tài liệu


本要件に対して、可能な限りServiceNowの標準機能を活用する方針とする。

申請の承認プロセスについては、標準のWorkflow機能により2段階承認を実現する。
また、承認状況の可視化については、標準のDashboard機能を利用することで対応可能である。

一方で、承認条件が複雑なケースについては、標準機能では対応が困難なため、
一部カスタマイズを実施することを検討する。

　- メールで業務プロセスが管理されている
　- 対応状況の可視化ができていない

■課題
　- 承認対応に遅延が発生している
　- 対応状況の可視化が不十分
　- 承認停止の原因が不明確

■要件
　- 承認フローを標準化すること
　- 一定金額を超える場合に承認条件を追加すること
　- 承認状況を可視化すること
　- 承認停止時の情報連携を行うこと

■対応方針
　- 保守性およびコスト観点から、ServiceNowの標準機能を優先的に利用する
　- 承認フローはFlow Designerにより実装し、金額条件による分岐は標準機能で対応する

　- 承認停止時の通知は、NotificationおよびSLA機能を組み合わせて実現する
　- 標準機能で対応困難な場合に限り、影響範囲を考慮した上で最小限のカスタマイズを実施する

■効果
　- 承認リードタイムの短縮
　- 承認状況の可視化による業務の透明性向上
　- 承認停止原因の明確化による対応迅速化
